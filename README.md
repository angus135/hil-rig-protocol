# HIL-RIG Protocol

Shared protocol, encoding, and compatibility definitions for the HIL-RIG project.

This repository defines the common interface between the HIL-RIG MCU firmware and the HIL-RIG Python API. It is intended to act as the single source of truth for the binary protocol used to communicate with the HIL-RIG hardware.

The repository is designed to be used by both:

- `hil-rig-mcu-firmware`
- `hil-rig-python-api`

The firmware should link against the C implementation directly. The Python API should use the same protocol definitions through bindings, generated code, or another mechanism that preserves compatibility with the C implementation.

> **Status:** Early project skeleton. Many protocol details are not implemented yet. Sections marked `TODO` are intentionally left incomplete until the protocol is formally defined.

---

## Purpose

The HIL-RIG system requires the host-side Python software and the MCU firmware to agree on the exact format of all data exchanged across the communication link.

This includes:

- Transport framing
- Packet boundaries
- Packet IDs
- Payload encoding
- Instruction formats
- Result formats
- Error codes
- Version compatibility
- Validation rules
- Golden test vectors

Rather than duplicating packet definitions separately in the firmware and Python API repositories, this repository provides a shared protocol layer used by both sides.

The main goal is to ensure that the bytes generated by the Python API are exactly the bytes expected by the MCU firmware, and that the data returned by the MCU firmware is decoded consistently by the Python API.

---

## Repository Role

This repository defines **bytes on the wire**.

It should contain protocol-level logic only. It should not contain hardware-specific firmware logic or high-level Python application behaviour.

### This repository should define

- Binary packet formats
- Transport framing
- Application-layer packet structure
- Instruction encoding and decoding
- Result encoding and decoding
- Protocol version definitions
- Error code definitions
- Validation rules
- Shared C encoder/decoder implementation
- Python bindings or binding support
- Golden vectors for cross-language compatibility testing
- Protocol documentation

### This repository should not define

- STM32 peripheral drivers
- FreeRTOS tasks
- DMA handling
- USB CDC driver internals
- GPIO port mappings
- ADC hardware configuration
- Timer register configuration
- Python plotting tools
- Python user-facing test builder APIs
- Firmware execution manager internals
- Hardware-specific scheduling behaviour

The firmware and Python API may have completely different internal structures. This repository only defines the shared protocol boundary between them.

## Architecture Overview

The protocol is split into several layers.

```text
Host Python API
    |
    | High-level test construction
    |
Python protocol bindings
    |
    | Shared encoding / decoding
    |
HIL-RIG protocol library
    |
    | Application packets
    |
Transport layer
    |
    | Framed byte stream
    |
USB / UART / other physical transport
    |
MCU firmware transport receiver
    |
    | Packet decoding
    |
Firmware application layer
    |
    | Instruction scheduling and execution
    |
Execution manager / hardware drivers
```

The main protocol areas are:

- Transport layer
- Application layer
- Instruction encoding
- Result encoding
- Versioning and compatibility

Detailed definitions for each area should eventually be moved into `docs/` once the protocol is stable enough to document formally.

## Public C API

The C API should be usable by both firmware and host-side bindings.

The API should avoid dynamic allocation where possible, especially for code that may be used in the MCU firmware.

Preferred style:

```c
HIL_Status HIL_EncodeInstruction(
    const HIL_Instruction *instruction,
    uint8_t *out_buffer,
    size_t out_buffer_len,
    size_t *bytes_written
);

HIL_Status HIL_DecodeInstruction(
    const uint8_t *buffer,
    size_t buffer_len,
    HIL_Instruction *out_instruction
);
```

### API Design Principles

The public C API should:

- Use explicit input and output buffers
- Avoid heap allocation in core encode/decode paths
- Return clear status/error codes
- Validate input lengths before reading
- Avoid undefined behaviour on malformed packets
- Be suitable for firmware use
- Be easy to wrap from Python
- Keep hardware-specific behaviour out of the shared library

## Include Path Convention

Public headers are namespaced under:

```text
include/hil_rig_protocol/
```

This means includes should use the full project namespace:

```c
#include "hil_rig_protocol/version.h"
#include "hil_rig_protocol/transport/frame.h"
#include "hil_rig_protocol/application/packet.h"
```

Avoid public includes such as:

```c
#include "transport/frame.h"
```

Using the project namespace reduces the chance of header name collisions when this repository is included as a submodule in the firmware or Python API repositories.

## Firmware Suitability

The protocol implementation should be suitable for embedded firmware use.

Core encode/decode paths should avoid:

- Heap allocation
- Recursion
- Unbounded loops
- Large stack allocations
- Hidden global state
- Blocking operations
- Hardware-specific assumptions

Preferred behaviour:

- Caller provides input and output buffers
- Function returns required buffer size when output buffer is too small, if practical
- All packet lengths are checked before access
- Malformed input is rejected safely
- Decode functions do not read beyond provided buffer length
- Encode functions do not write beyond provided output buffer length

## Protocol Compatibility Contract

This repository should be treated as the compatibility contract between the Python API and firmware.

Any change that affects bytes on the wire should be considered a protocol change.

Examples of protocol-breaking changes:

- Changing packet field order
- Changing field width
- Changing enum values
- Changing CRC calculation
- Changing byte order
- Changing payload length rules
- Removing packet types
- Changing instruction timing semantics
- Changing result encoding

Examples of potentially non-breaking changes:

- Adding internal helper functions
- Refactoring implementation without changing encoded bytes
- Adding new packet types while preserving old ones
- Adding new error codes while preserving old ones
- Adding new optional fields only if older implementations can safely ignore them

Everything else belongs in either the firmware repository or the Python API repository.
